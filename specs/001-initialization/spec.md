# Feature Specification: Stock Analyzer Platform — Initialization

**Feature Branch**: `001-initialization`
**Created**: 2026-04-22
**Status**: Draft
**Input**: User description: "本项目是一个基于 Next.js 的股票数据分析网站，包含首页、邀请码注册登录、自选股、Dashboard（午评/晚评/未来预测、启动在即、主升浪、自选股、预测推荐、个股 AI 分析菜单）、新闻模块；股票数据经 Sealos 代理的 AKShare + AKTools 每日 11:30 与 15:15 全量抓取、仅保留 60 天；金融资讯经财联社 CLS（AkShare `stock_telegraph_cls`）随股票数据抓取并在 21:00 与次日 06:00 额外抓取、保留 7 天；AI 摘要 / 午评 / 晚评永久保留；界面使用 MagicUI（Number Ticker、Marquee、Bento Grid）增强。"

## Clarifications

### Session 2026-04-22

- Q: 未来预测 / 预测推荐中的"板块"采用哪种粒度？（申万一级行业 / 概念板块 /
  两者兼用 / 中信行业） → A: **概念板块（题材板块）**，数据源 AkShare
  `stock_board_concept_*` 系列。
- Q: 自选股数量上限与 Dashboard 展示策略？（硬上限 50 全量 / 硬上限 100
  显示 20 可展开 / 无硬上限 + 虚拟滚动 / 硬上限 30 全量） → A: **无硬上限**，
  Dashboard 自选股板块使用虚拟滚动承载任意数量；当用户自选数量 ≥ 200 时
  显示非阻塞软提示"自选过多可能影响浏览体验，建议精简"。
- Q: 账号注销 / 数据导出是否在本期实现？（不做 / 只做删除账号 / 删除 + 自选
  CSV 导出 / 完整 GDPR 风格导出） → A: **本期不实现**；用户注销 / 数据清理
  延后到下一个"管理后台"feature，由管理员入口统一处理，理由是本产品为
  邀请码受控的小圈子，无 GDPR / CCPA 辖区压力。
- Q: 登录失败 / 暴力破解防护？（不做 lockout / 账号级 5 次软速率限制 /
  账号级 10 次硬锁定 / 账号 + IP 双重速率限制） → A: **不做 lockout**；
  登录失败仅返回"邮箱或密码错误"的友好提示，不做速率限制也不做账号锁定。
  这是经评估后接受的 MVP 残留风险——因为用户群为邀请码受控的小圈子、邮箱
  非公开发布、且账号数量有限；若后续实际暴露面扩大，再以独立 feature 引入
  速率限制。
- Q: A 股休市横幅"今日关闭"的持久化口径？（自然日边界 / A 股交易日边界 /
  数据指纹变化时重置） → A: **自然日边界**（Asia/Shanghai 当日 00:00
  重置）。localStorage key 形如 `banner-dismissed-{YYYY-MM-DD}`，与 A 股交易
  日历完全解耦；每个自然日最多提醒一次，长假期间每天各提醒一次以强调"今天
  依然休市"。
- Q: 初始化历史数据回填策略？（不回填 / 一次性回填 60 日 / 延后 60 日启用
  筛选板块 / 仅回填 15 日折衷） → A: **上线首日一次性回填最近 60 个 A 股
  交易日的全量行情数据**（AKShare `stock_zh_a_hist()` 历史接口），写入
  StockDaily；该回填任务在常规 11:30 / 15:15 抓取启动**之前**执行且只执行
  一次，以保证 Day-1 所有筛选 / 累计涨跌幅板块即可工作。
- Q: 账号名允许的字符集与长度？（ASCII+_-3~20 / 允许 Unicode+slug /
  ASCII 纯字母数字 / 中英文混合禁 emoji） → A: **仅允许 ASCII 字母 + 数字
  + `_` + `-`，长度 3~20 字符**（正则 `^[A-Za-z0-9_-]{3,20}$`）。账号名仅作为
  站内唯一显示名，不再作为登录凭证；登录凭证为真实邮箱。中文用户请使用拼音或
  英文名作为账号名。
- Q: Dashboard 数据实时性 / 轮询策略？（不自动刷新 / Supabase Realtime 推送
  / 60s 轮询 + 手动按钮 / 仅交易时段轮询） → A: **服务端抓取完成后通过
  Supabase Realtime 推送 `data-updated` 事件，前端按板块粒度局部刷新**；
  不再做客户端定时轮询。Dashboard 右上角仍保留"手动刷新"按钮兜底。
- Q: Dashboard 卡片可见性与用户偏好？（完全固定 / 允许隐藏不可重排 / 完全
  可个性化 / 仅 AI 三栏可隐藏） → A: **完全固定布局**，所有卡片统一顺序、
  不可隐藏、不可重排；Bento Grid 仅负责响应式断点适配。个性化能力延后到
  未来独立 feature，本期**禁止**引入卡片 per-user 可见性 / 顺序字段。

### User Story 1 — 邀请码注册与登录闭环 (Priority: P1)

访客访问首页，在明显位置看到"登录 / 注册"入口；点击后在首页上层弹出一个半透明弹窗，
凭固定邀请码 `violet-everGarden` 使用真实邮箱、自选账号名与密码完成注册，并通过
邮箱确认激活账号；已激活用户用邮箱 + 密码登录。登录时可勾选"7 天内免登录"。
登录成功或邮箱确认回跳建立会话后，页面自动跳转至 Dashboard。
任何未登录访客试图直接访问 Dashboard、自选股、新闻等非首页路由时，浏览器会被
重定向回首页并自动打开注册/登录弹窗，已登录用户则可以自由访问。

**Why this priority**: 没有可用的身份通道，后续任何"个性化"模块（自选股、AI 分析、
未来预测推荐）都失去意义；邀请码机制是阻止陌生流量、保持"小圈子使用"定位的唯一
手段。本故事独立完成即交付"受控访问的 Next.js 骨架"这一价值。

**Independent Test**: 使用一台干净的浏览器，依次验证以下路径即可：
(1) 未登录访问 `/dashboard` → 被拽回首页并弹出登录弹窗；
(2) 用错误邀请码注册 → 拒绝；
(3) 用正确邀请码 + 真实邮箱 + 新账号名 + 密码 → 收到确认邮件，点击确认后进入
Dashboard；
(4) 勾选"7 天免登录"登出、7 日内再次打开网站 → 直接处于已登录状态；
(5) 不勾选"7 天免登录"登出后关闭浏览器会话 → 下次打开仍需登录。

**Acceptance Scenarios**:

1. **Given** 访客首次访问根路径，**When** 点击首页的"注册"按钮，**Then**
   在首页上方浮出带 MagicCard 样式的半透明弹窗，注册态包含邀请码 / 邮箱 /
   账号名 / 密码四个字段以及"注册"与"已有账号去登录"两个按钮。
2. **Given** 弹窗中邀请码输入非 `violet-everGarden`，**When** 点击"注册"，
   **Then** 表单在同一弹窗内给出"邀请码无效"的行内错误，不发起账号创建，
   可再次尝试。
3. **Given** 邀请码正确、邮箱格式有效且邮箱 / 账号名均未被占用，**When** 点击
   "注册"，**Then** 账号在后端生成、Supabase 发送确认邮件，弹窗提示用户查收
   邮箱；用户点击确认链接后回到站点并进入 Dashboard。
4. **Given** 已确认邮箱的用户在登录弹窗输入正确邮箱和密码并勾选"7 天免登录"，**When**
   点击"登录"，**Then** 会话记忆至少保持 7 × 24 小时，期间关闭浏览器再打开
   仍自动进入 Dashboard。
5. **Given** 未登录访客直接访问 `/dashboard`、`/watchlist`、`/news`
   任意非首页路由，**When** 浏览器完成导航，**Then** 被重定向回首页且自动
   打开登录弹窗，原始目标路径被记录以便登录后回跳。
6. **Given** 同一邀请码 `violet-everGarden` 已被其它用户使用过，**When** 新
   访客再次用该邀请码注册，**Then** 仍然接受（邀请码可无限次复用）。

### User Story 2 — 股票数据每日定时抓取与 60 天滚动保留 (Priority: P1)

系统每个交易日自动在中午 11:30（午盘收盘后）与下午 15:15（全天收盘后）各执行一次
全量 A 股股票数据抓取，数据经 Sealos 代理的 AKShare + AKTools 服务获取后写入持久化
存储。系统仅保留最近 60 个交易日的行情数据，更早的数据在每次抓取后被自动清理。

**Why this priority**: 后面所有筛选（启动在即 / 主升浪）、自选股涨跌幅、AI 点评
都建立在"今天有一份较新的收盘价"上；调度失败或数据陈旧会连锁影响所有看板。

**Independent Test**: 可独立验证：
(1) 在 11:30 与 15:15 后各 5 分钟内，检查最新一条行情的 `trade_date` 是否为
    当前交易日；
(2) 在数据库中插入一条 `trade_date = today - 61 交易日` 的行，触发下一次抓取后，
    该行被清理；
(3) 若抓取失败，系统记录失败事件、保留上一份数据、并在产品内给出"数据截止至
    YYYY-MM-DD HH:mm"的提示。

**Acceptance Scenarios**:

1. **Given** 当前为交易日 11:30 到 11:35 之间，**When** 调度任务触发抓取，
   **Then** 系统获取当日午盘全量股票数据（至少包含股票代码、名称、市场板块、
   是否 ST、总市值、当日开 / 高 / 低 / 收、当日成交量），并写入行情表；
   最新 `fetched_at` 时间戳更新。
2. **Given** 抓取完成时存储中存在 `trade_date < today - 60 交易日` 的行情数据，
   **When** 抓取成功事务提交，**Then** 这些超出 60 交易日窗口的行情数据在同一
   事务或紧随的清理任务中被删除。
3. **Given** Sealos 代理或上游 AKShare 返回错误，**When** 抓取任务捕获异常，
   **Then** 已有的最近一次成功数据保持原样；系统记录失败事件与原因，Dashboard
   在显著位置以非阻塞文案提示"行情数据截止至 <上次成功时间>"。
4. **Given** 非交易日（周末或节假日），**When** 调度器到达 11:30 / 15:15，
   **Then** 任务可以选择跳过，或执行但未获得新的交易日数据；两种情况都不应
   覆盖掉上一交易日的正常数据。
5. **Given** 项目首次部署到生产环境且 StockDaily 为空，**When** 应用启动，
   **Then** 系统 **必须** 先执行一次性历史回填任务，通过 AKShare 历史接口
   拉取最近 60 个 A 股交易日的全量行情数据写入 StockDaily；回填完成之前
   **不得**启动 11:30 / 15:15 的常规调度，**不得**开放 Dashboard 上的
   启动在即 / 主升浪 / 自选股 5 日累计涨跌幅板块（应以"系统初始化中"
   友好占位替代）。回填只执行一次，下次部署检测到 StockDaily 非空或
   `backfill_completed` 标记存在时跳过。

### User Story 3 — Dashboard 核心筛选板块：启动在即 + 主升浪 (Priority: P1)

登录用户进入 Dashboard 后，能够在两个并列的股票列表中看到系统基于最近一次收盘
数据自动筛选出的：
(1) **启动在即** —— 当前处于震荡末端、当日放量突破、具备开启主升浪潜力的个股；
(2) **主升浪** —— 已经确认进入主升浪、适合右侧交易的个股。
每条记录显示股票名称（左）与当日涨跌幅（右），涨为红、跌为绿；点击任一条目可
进入 User Story 7 描述的 AI 弹出菜单。

**Why this priority**: 这是本产品相对于通用行情网站的核心差异化价值。即便 P2、P3
模块全部未交付，用户每天登录只为查看这两份名单也成立。

**Independent Test**: 使用 User Story 2 抓取得到的真实（或 fixture）数据，直接
触发筛选流水线，校验两份名单：
(1) 所有返回的代码都属于 A 股主板或科创板，非 ST，总市值 > 100 亿；
(2) "启动在即"名单中的每一只都满足：当日涨幅 > 4%、当日成交量 > 过去 10 个
    交易日成交量均值、当日收盘价 > 10 日均线、过去 11 个交易日（允许忽略最后
    1~2 个交易日）价格处于区间震荡状态；
(3) "主升浪"名单中的每一只都满足：当日收盘价 > 5 日均线、周线最新收盘 >
    5 周均线、当日收盘价 > 10 日均线、过去 5 个交易日整体呈连续上涨趋势
    （允许小幅回撤但总体趋势向上）。

**Acceptance Scenarios**:

1. **Given** 已有最新一份成功抓取的行情数据，**When** 用户打开 Dashboard，
   **Then** "启动在即"与"主升浪"两个列表在 2 秒内展示结果（或先展示骨架屏，
   实际数据就绪后替换），且两份列表各自最多展示 **20** 条（数量不足时全部
   展示，零条时显示"今日无符合条件的标的"）。
2. **Given** 某只个股属于中小板、创业板、北交所或 ST / \*ST / 退市股，
   **When** 筛选执行，**Then** 该股票一定不会出现在两份名单中。
3. **Given** 某只股票总市值低于 100 亿元（人民币），**When** 筛选执行，
   **Then** 该股票不会出现在两份名单中。
4. **Given** "启动在即"候选股在过去 11 个交易日的价格波动超过既定震荡阈值
   （详见 Assumptions 对区间震荡的量化定义），**When** 筛选执行，**Then**
   该股不会入选"启动在即"，即便其它条件全部满足。
5. **Given** 用户点击列表中的任一股票条目，**When** 交互发生，**Then** 在
   交互首帧内必须看到可见反馈（hover/按下态 → 菜单浮出），不允许出现"点击
   后 UI 无变化、后台静默请求"的反模式（闭环宪法原则五）。

### User Story 4 — 自选股管理与 Dashboard 自选股板块 (Priority: P2)

登录用户可以通过股票代码或股票名称搜索并添加个股到自选列表、在自选股页面通过
拖拽改变顺序，或删除某只自选股。Dashboard 上专门有一个"自选股"板块，实时展示
当前用户的自选股及其**当日涨跌幅**与**5 日累计涨跌幅**，顺序与用户在管理页面
配置的顺序保持一致，跨路由切换后顺序与内容保留。

**Why this priority**: 自选股能立刻把产品变成"我的行情站"，但其本身并不依赖
启动在即 / 主升浪的筛选结果，可以作为 P1 之上第一个增强价值的独立切片。

**Independent Test**:
(1) 从空自选开始，依次搜索"平安银行"和 `000001`，两次输入都能命中同一只股票
    并加入列表；重复添加被合并为一条；
(2) 拖拽一只股票到头部，刷新页面后顺序保持；
(3) 在 Dashboard 的自选股板块能看到刚才加入的股票、当日涨跌幅与 5 日累计涨跌幅。

**Acceptance Scenarios**:

1. **Given** 用户在自选股页面的搜索框输入 6 位股票代码，**When** 匹配到唯一股票，
   **Then** 该股票出现在搜索结果中，点击"加入自选"即被追加到列表末尾。
2. **Given** 用户输入中文股票名称关键字（至少 2 个汉字），**When** 系统在已有
   行情数据中进行名称前缀/包含匹配，**Then** 最多返回前 10 条结果供选择。
3. **Given** 自选列表已有至少 2 只股票，**When** 用户通过拖拽把某只股票移到
   任意其它位置，**Then** 放手瞬间列表顺序即时更新（乐观 UI），后端持久化
   成功后状态保持；若持久化失败，回滚到拖拽前的顺序并提示失败原因。
4. **Given** 用户切换到 Dashboard，**When** 自选股板块渲染，**Then** 板块
   展示顺序、个数与自选页面完全一致，每一条显示当日涨跌幅与 5 日累计涨跌幅，
   涨为红、跌为绿。
5. **Given** 用户在 Dashboard 与自选股页面之间多次切换路由，**When** 回到任一
   页面，**Then** 自选列表不会被重新"从零拉取"造成可见闪烁（走系统级 Zustand
   store + 请求缓存，闭环宪法原则七）。
6. **Given** 用户自选数量达到 200 只或以上，**When** 用户进入自选管理页面
   或 Dashboard 自选股板块，**Then** 顶部出现非阻塞软提示"自选过多可能影响
   浏览体验，建议精简"；用户仍可继续添加（无硬上限），列表在虚拟滚动下保持
   流畅（滚动帧率 ≥ 50 fps，DOM 中保留的卡片节点数 ≤ 60 个，不随总数增长）。

### User Story 5 — Dashboard 三栏 AI 点评：今日午评 / 今日晚评 / 未来预测 (Priority: P2)

Dashboard 顶部或显著位置并列展示三块 AI 生成的点评内容：
**今日午评**（午盘后更新一次）、**今日晚评**（全天收盘后更新一次）、
**未来预测**（每日晚间随新闻额外抓取后生成一次）。三块内容均通过 DeepSeek LLM 基于
当日行情快照与当日金融资讯生成，并永久保留历史。

**Why this priority**: 点评内容给用户"专业分析陪伴感"，是拉长停留时长的核心
体验，但在没有任何 AI 输出前，前面 3 个 P1 故事已构成最小可用产品。

**Independent Test**:
(1) 在 11:30 抓取完成后，"今日午评"在 10 分钟内更新，且内容与当日整体行情一致；
(2) 当日 21:00 资讯抓取后，"未来预测"更新，且内容明显引用了当日资讯信息；
(3) 过去任一历史日期的午评 / 晚评 / 未来预测可通过产品内入口（或至少通过数据
    库查询）回溯到。

**Acceptance Scenarios**:

1. **Given** 当日 11:30 的股票数据抓取成功，**When** 点评生成任务触发，
   **Then** 在 10 分钟内生成"今日午评"文本并在 Dashboard 对应卡片展示；
   卡片页脚 **必须** 同时显示两条时间戳：(a) **AI 更新时间**（本次 AI 调用
   完成的本地时间），(b) **参考数据时间**（生成时所引用的行情快照交易日 +
   时段，例如"2026-04-22 11:30 午盘"）。内容包括当日市场总体涨跌、活跃板块、
   值得关注的动向。
2. **Given** 当日 15:15 的股票数据抓取成功，**When** 点评生成任务触发，**Then**
   在 10 分钟内生成"今日晚评"，并同步保存；已有的"今日午评"内容保持不变；
   "今日晚评"卡片页脚同样展示 AI 更新时间 + 参考数据时间（例如"2026-04-22
   15:15 尾盘"）。
3. **Given** 当日 21:00 或次日 06:00 的资讯额外抓取完成，**When** 预测生成触发，
   **Then** 在 10 分钟内生成"未来预测"；预测内容涉及的板块与 User Story 8
   "预测推荐"共用同一份判断；"未来预测"卡片页脚展示 AI 更新时间 + 参考资讯
   批次时间（例如"2026-04-22 21:05 新闻批次"）。
4. **Given** DeepSeek LLM 接口在某次调用中失败或超时，**When** 生成任务观测到失败，
   **Then** 该卡片保留上一次的成功内容并叠加"更新失败，已回退至上一版本"的
   小字提示；**AI 更新时间与参考数据时间仍显示上一次成功的值**，不被清空；
   下次调度时间到达后自动重试。
5. **Given** 任一三栏对应的 AI 文本已被成功生成并落库，**When** 行情 / 资讯
   数据被后续的 60 天 / 7 天滚动清理触达，**Then** AI 生成的点评 / 预测内容
   本身**不会**被清理；但该条目的"参考数据时间"仍以生成当时捕获的值保留，
   不会因原始数据清理而丢失。
6. **Given** 某一块 AI 卡片的参考数据时间距离当前时刻超过该卡片对应的"新鲜度
   阈值"（详见 Assumptions 关于各 AI 产物新鲜度阈值的默认值），**When** 用户
   打开 Dashboard，**Then** 该卡片必须在标题或页脚附近显示明显的**过期标识**
   （例如 `已过期` 徽章、琥珀色色点、或"上次更新 XX 小时前"等文案），提示
   用户内容可能不再反映最新情况。

### User Story 6 — 新闻模块：24h 聚合 + AI 总结 (Priority: P2)

新闻页面展示过去 24 小时内国内外重要财经新闻的完整列表；顶部固定一段由 AI 基于
同一批新闻生成的总结段落，帮助用户快速了解"这 24 小时发生了什么"。新闻数据
只在本地保留近 7 天，过期的原始资讯会被滚动清理。

**Why this priority**: 新闻是 AI 点评 / 未来预测的上游语料，同时本身也是用户的
独立浏览入口；但它独立于 P1 的筛选板块，可以作为 P2 切片单独交付。

**Independent Test**:
(1) 打开新闻页面时，页面顶部存在一段不超过 300 字的 AI 总结；
(2) 下方列表至少展示过去 24 小时内的资讯条目；
(3) 滚动到较早的条目（> 24 小时但 ≤ 7 天）不会出现在主视图中，但可以通过
    "查看近 7 天"入口加载到；
(4) 超过 7 天的原始资讯在存储中已被清理。

**Acceptance Scenarios**:

1. **Given** 当日任意一次新闻抓取完成（股票数据同步、21:00、次日 06:00
   共三类触发），**When** 新闻抓取任务结束，**Then** 触发 AI 总结生成，
   新生成的总结替换顶部已显示的总结内容，同时保留旧版本到历史表供回溯。
2. **Given** 用户进入新闻页面，**When** 列表加载，**Then** 默认展示过去
   24 小时内按发布时间倒序排列的全部资讯，每条至少包含标题、来源、发布时间、
   正文摘要；点击某条可查看完整正文。
3. **Given** 存储中存在 `published_at < now - 7 天` 的新闻原文，**When**
   下一次抓取任务结束，**Then** 这些过期新闻被清理；其对应被 AI 引用过的
   总结文本**不会**被一并删除。
4. **Given** 财联社（AkShare `stock_telegraph_cls`）接口调用失败或返回空，
   **When** 抓取任务捕获异常，**Then** 既有新闻保持原样，Dashboard / 新闻
   页面以小字提示"资讯更新失败"，并在下一调度窗口重试。

### User Story 7 — 个股弹出菜单：AI 个股介绍 + AI 股票分析 (Priority: P2)

在 Dashboard 的任一股票列表（启动在即 / 主升浪 / 自选股 / 预测推荐）中点击
某只个股条目，会在点击位置附近弹出一个类似右键菜单样式的浮层，提供两个功能：
**AI 个股介绍**（所处行业、主营业务、竞争优势概览）与 **AI 股票分析**（近期
走势、量价特征、可能的风险点）。点击任一功能后，在另一弹窗中展示 AI 生成的
对应内容。

**Why this priority**: 本故事让用户从"看名单"升级为"研究名单"，是停留时长和
复购感的关键增量，但独立于 P1 筛选功能的存在性。

**Independent Test**:
(1) 在任一列表中单击一只股票 → 首帧内看到菜单浮出；
(2) 点击"AI 个股介绍" → 弹窗内出现该股票的行业 / 主营 / 竞争优势段落；
(3) 点击"AI 股票分析" → 弹窗内出现该股票近期走势分析段落；
(4) 两种分析文本同一日内可以缓存复用（同一股票同一日再次点击无需重新请求）。

**Acceptance Scenarios**:

1. **Given** 用户在 Dashboard 任一股票列表中点击一只个股条目，**When** 单击
   发生，**Then** 该条目上或旁边立即弹出菜单浮层，包含"AI 个股介绍"与
   "AI 股票分析"两个选项；在弹出前不允许存在任何"空白等待 > 100 ms"的间隙
   （闭环原则五）。
2. **Given** 用户选择"AI 个股介绍"，**When** 系统已在当日或历史上生成过该股票
   的介绍，**Then** 直接从缓存返回、弹窗内即时展示；否则进入 pending 态并在
   LLM 返回后替换内容，失败时显示"暂时无法生成介绍，请稍后重试"。
3. **Given** 用户选择"AI 股票分析"，**When** LLM 返回完成，**Then** 弹窗中
   展示结构化文本（近期走势 + 量价特征 + 风险提示三段），并附上生成时间戳。
4. **Given** 菜单已弹出时用户点击菜单外区域或按 `Esc`，**When** 交互发生，
   **Then** 菜单关闭且不触发任何请求。

### User Story 12 — 多市场休市状态提示与 A 股休市数据回退 (Priority: P2)

登录用户进入 Dashboard 时，系统根据当日（北京时间）A 股交易日历自动判定休市状态：
(1) 当 A 股当日休市（周末 / 法定节假日），在 Dashboard 顶部显示一条非阻塞的横幅，
文案形如"今日 A 股休市，数据基于上一个交易日 &lt;YYYY-MM-DD&gt;"；
(2) 同一时刻，Dashboard 上所有依赖 A 股行情的板块（启动在即、主升浪、自选股涨跌、
AI 午评 / 晚评 / 未来预测、预测推荐、核心指数 Ticker）的数据口径统一回退到
**最近一个 A 股交易日**，而不是显示空数据或"无数据"占位；
(3) Dashboard 上新增一张**市场状态卡片**（无论 A 股是否休市，始终显示），独立
列出 A 股、美股、港股、日股、韩股五个市场在"当前北京时间所属自然日"的开市 /
休市状态，每个市场一行，以文字 +  色点 / 图标同时标识。

**Why this priority**: 若不处理，周末或节假日首次打开 Dashboard 的用户会看到
"今日涨跌幅"、"启动在即"等板块引用的是上一个交易日数据但没有任何说明，极易误判
为"今日行情"。这是一次对信息可信度的关键保障；但它构建在 P1（抓取 + 核心筛选）
之上，在 P1 全部成立之前无法独立展示价值，因此列为 P2。

**Independent Test**: 通过切换系统时间（或 mock 交易日历）可独立验证以下路径：
(1) 在周六打开 Dashboard → 顶部出现"今日 A 股休市"横幅，引用日期为上周五；
(2) 启动在即、主升浪、自选股涨跌、AI 三栏点评四处引用数据的交易日 **均 = 上周五**；
(3) 市场状态卡片显示：A 股休市 / 美股休市 / 港股休市 / 日股休市 / 韩股休市；
(4) 切回工作日 → 横幅消失、数据恢复为当日；
(5) 切到一个 A 股正常交易但美股因独立日休市的日期 → 横幅不出现，卡片仅美股行
显示休市。

**Acceptance Scenarios**:

1. **Given** 当前北京时间所属自然日属于 A 股休市日（周末或法定节假日），
   **When** 登录用户打开 Dashboard，**Then** 在顶部非阻塞区域 1 秒内展示横幅，
   文案包含"今日 A 股休市"与被引用的上一个 A 股交易日日期；横幅不遮挡主内容
   操作、提供"关闭"按钮，关闭状态以 Asia/Shanghai 自然日边界持久化（每天
   00:00 重置，长假期间每自然日各提醒一次）。
2. **Given** 当日 A 股休市，**When** Dashboard 加载启动在即 / 主升浪 / 自选股
   涨跌 / AI 午评 / AI 晚评 / 未来预测 / 预测推荐 / 核心指数 Ticker 任一板块，
   **Then** 板块展示的数据**必须**来自"最近一个 A 股交易日"，且在各板块上以
   小字显式标注数据日期（不强制每块都显示，但**至少** Dashboard 顶部的横幅
   里出现一次、各列表 header 有可悬停提示）。
3. **Given** 当日 A 股正常交易，**When** 用户打开 Dashboard，**Then** 顶部横幅
   不出现；仅当其它市场（美 / 港 / 日 / 韩）有休市时，它们的休市状态只通过市场
   状态卡片呈现，不额外弹横幅。
4. **Given** 用户打开 Dashboard 的任意时刻，**When** 市场状态卡片渲染，**Then**
   卡片按固定顺序（A 股 → 美股 → 港股 → 日股 → 韩股）列出五个市场，每行以
   "市场名 + 开市 / 休市 + 视觉标识（色点或图标）"呈现；颜色使用与"涨红跌绿"
   不冲突的语义色（建议开市为中性绿 / 蓝、休市为中性灰）。
5. **Given** A 股休市日的调度器触发时刻，**When** 系统判定当日为 A 股休市，
   **Then** 11:30 与 15:15 的行情抓取任务被跳过或空跑，**不得**覆盖既有的最近
   一个 A 股交易日的数据；审计日志记录"skipped: A-share holiday"。

### Edge Cases

- **未登录 + 深链接**：用户通过外部分享链接直接打开 `/news/xxx` 或某只股票
  详情页，应记录原始路径，登录成功后自动回跳到该路径，而不是只停留在
  Dashboard。
- **邀请码字段被粘贴多余空白 / 大小写变形**：系统必须对 `violet-everGarden`
  的比较采用严格一致（区分大小写）策略；前端在输入后应 trim 两端空白再比对，
  避免"看起来一样、比对失败"的挫败。
- **账号名含不可见 / 相似字符**：注册表单对账号名 `必须` 先 trim 两端空白
  并拒绝任何非 ASCII 字母 / 数字 / `_` / `-` 的输入（见 FR-008）；特别是
  对全角破折号、零宽空格、相似西里尔字母等"视觉相同但编码不同"的字符
  一律拒绝，避免未来账号名伪造 / 冒充。
- **邮箱格式与确认状态**：注册表单必须拒绝明显非法邮箱；未完成邮箱确认的用户
  尝试登录时，系统必须提示"请先完成邮箱确认"，不得静默失败。
- **注册并发**：同一邮箱或账号名在极短时间内被两次注册请求命中时，后一次必须
  返回"邮箱已注册"或"账号已存在"，且不会意外复写第一次的密码散列。
- **会话劫持抵御**：选择"7 天免登录"时，服务器签发的长期凭证必须具备
  `HttpOnly`、`Secure`、`SameSite` 保护；任何"登出"操作必须同时使长期凭证失效。
- **登录暴力破解的残留风险**：本期明确不引入 rate-limit / lockout（见 FR-007
  与 Assumptions），这是一次对"安全 vs 误伤用户"的主动权衡。失败事件仍会
  完整落审计日志；若审计日志在短期内观察到同一邮箱失败次数异常（例如 1 小时
  内 > 30 次），由人工介入而非自动锁定。
- **非交易日 / 停牌**：停牌股票在抓取当天没有成交量，涨跌幅口径应以"前一个
  有效交易日的收盘价"为准；停牌股不应出现在"启动在即"或"主升浪"中。
- **数据清理与筛选的时序**：60 天滚动清理必须在当次抓取成功之后再执行；
  不能先删后抓，否则上游失败会留下数据空洞。
- **前 1~2 日数据剔除带来的倒推窗口不足**：当最新数据不足以回看 11 个交易日
  （例如系统上线不久），"启动在即"结果集必须为空并给出"样本不足，暂无筛选
  结果"的提示，而不是基于残缺数据得出误判。
- **DeepSeek LLM 出现有害 / 越界内容**：生成失败或输出越界时，前端展示
  "内容不可用，请稍后重试"，不把原始错误内容呈现给用户。
- **新闻 / 行情抓取同时压向 Sealos**：两类任务被调度到同一分钟时需要串行化，
  避免 Sealos 实例连接数暴涨。
- **自选股被上游下架 / 退市**：若某自选股在抓取中不再出现在行情全量表中，
  产品内必须以"暂无行情"占位显示而不是抛错；用户仍可手动从自选列表中删除它。
- **拖拽排序失败回滚**：乐观 UI 在后端保存失败时必须严格回滚顺序并用
  toast 告知失败原因，禁止"前端看起来成功、实际未存"。
- **多市场节假日交叉**：某日 A 股正常交易但美股因独立日 / 感恩节 / 圣诞节
  等休市，只应在市场状态卡片中把美股一行标为"休市"，**不**触发顶部横幅；
  反向同理（A 股休市但美股正常交易），横幅只由 A 股状态决定。
- **时区边界**：五个市场"今日是否休市"统一以 **Asia/Shanghai 所属自然日**
  为准，不按各市场本地时区；例如周六凌晨 1 点北京时间（周五美东晚间），即便
  美股仍在当地周五的交易时段，也视为"北京时间周六 = 美股休市"。
- **交易日历陈旧 / 未覆盖当前日期**：当市场交易日历数据过期或不覆盖当前日期
  时，Dashboard 顶部 / 市场状态卡片上必须以小字提示"日历已过期，请尽快更新"，
  但不阻塞用户访问；后台自动触发日历刷新。
- **AI 卡片过期标识的视觉语义**：过期标识 **不得** 仅以颜色区分新鲜 / 过期
  （例如仅用"绿 = 新鲜，琥珀 = 过期"），必须同时附带文字或图标，保证色盲
  用户和夜间模式下都能辨识。
- **AI 回退期间的时间戳显示**：详见 FR-053 末段——当 LLM 调用失败触发"回退
  至上一版本"时，卡片上展示的 `generated_at` 与 `source_data_at` **必须**
  保持为上一次成功的值，**禁止**显示为本次失败尝试的时间（避免用户误判为
  刚刚有过一次成功更新）。
- **A 股休市当日的 AI 点评 / 未来预测调度**：午评 / 晚评在 A 股休市日不生成
  新版本（避免基于陈旧行情产出误导文本）；"未来预测"仍按 21:00 / 06:00 正常
  生成，但其引用的行情上下文来自最近一个 A 股交易日。

## Requirements *(mandatory)*

### Functional Requirements

**身份与访问控制**

- **FR-001**: 系统 **必须** 在首页提供注册与登录两个操作入口，两者均通过
  同一个半透明弹窗承载，弹窗使用 MagicUI MagicCard 风格。
- **FR-002**: 注册 **必须** 同时校验邀请码、真实邮箱、账号名、密码；邀请码必须
  严格等于 `violet-everGarden`（区分大小写、前后端各自 trim 两端空白），且同一
  邀请码 **必须** 支持无限次复用。邮箱必须通过格式校验、trim 后小写规范化，并
  作为 Supabase Auth 的主登录凭证。
- **FR-003**: 系统 **必须** 以散列方式存储用户密码，不得明文保存或以可逆方式
  加密。
- **FR-004**: 登录 **必须** 提供"7 天免登录"选项；勾选时服务器签发的会话凭证
  有效期 ≥ 7 × 24 小时，未勾选时会话在浏览器会话结束时失效。
- **FR-005**: 系统 **必须** 对未登录用户访问任何非首页、非公共资源的路由进行
  拦截，重定向回首页并自动展开登录弹窗，同时记录原始目标路径以便登录成功
  后回跳。
- **FR-006**: 用户登出 **必须** 同时使短期会话与"7 天免登录"长期凭证双双失效。
- **FR-007**: 登录失败时，系统 **必须** 返回**统一的通用错误文案**"邮箱或
  密码错误"（不区分"邮箱不存在"与"密码错误"，避免账号枚举）；本期**不**
  对登录尝试实施账号级或 IP 级速率限制、也**不**启用账号锁定（经 Clarify
  确认，2026-04-22）。失败事件仍 **必须** 落审计日志（时间戳 + 规范化邮箱 +
  IP + user-agent），便于事后排查异常。
- **FR-008**: 账号名 **必须** 满足正则 `^[A-Za-z0-9_-]{3,20}$`（ASCII 字母 /
  数字 / 下划线 / 中划线，长度 3~20）。注册接口在接收提交后、在服务端以
  该正则作为唯一校验来源（**禁止**仅依赖前端校验），不符合时返回行内错误
  "账号名仅支持字母、数字、下划线、中划线，长度 3~20"；中文或其它非 ASCII
  字符一律拒绝。账号名仍按大小写不敏感判唯一（`Alice` 与 `alice` 不可
  同时存在，见 Assumptions）；入库时 **必须** 以小写形式规范化。账号名仅作为
  站内显示名与资料键，不作为登录凭证。
- **FR-009**: Supabase Auth **必须** 开启邮箱确认；注册完成后，系统必须提示用户
  查收确认邮件。邮箱未确认前不得视为已登录，登录时必须给出"请先完成邮箱确认"的
  可理解提示。

**自选股**

- **FR-010**: 用户 **必须** 能够通过股票代码（6 位数字）或股票名称（中文关键字
  ≥ 2 字，支持包含匹配）搜索并添加个股到自选列表；对同一股票的重复添加必须
  被合并为一条。
- **FR-011**: 用户 **必须** 能够通过拖拽对自选列表进行自定义排序；拖拽放手瞬间
  顺序即时更新（乐观 UI），后端持久化失败时必须回滚到拖拽前顺序。
- **FR-012**: 用户 **必须** 能够从自选列表中删除单只或多只股票。
- **FR-013**: 每位用户的自选列表 **必须** 在会话之间持久化，且在不同设备的
  相同账号间保持一致。
- **FR-014**: 自选股数量 **无硬上限**；当用户自选数量 ≥ **200 只** 时，自选
  管理页面与 Dashboard 自选股板块 **必须** 在顶部显示非阻塞软提示"自选过多
  可能影响浏览体验，建议精简"，提示 **不得** 阻塞后续添加，但应持续可见
  直至数量降至 200 以下。

**股票数据抓取与保留**

- **FR-020**: 系统 **必须** 在每个交易日 11:30（午盘）与 15:15（尾盘）各执行
  一次全量 A 股行情抓取；抓取的数据源为 Sealos 代理部署的 AKShare + AKTools。
- **FR-021**: 抓取的数据 **必须** 至少包含股票代码、名称、市场板块（主板 /
  科创 / 其他）、是否 ST、总市值、当日开 / 高 / 低 / 收、当日成交量、交易日期。
- **FR-022**: 系统 **必须** 仅保留最近 60 个交易日的行情原始数据，超过 60 个
  交易日的数据在每次抓取成功后的同一流水线中被清理。
- **FR-023**: 抓取失败 **不得** 覆盖既有成功数据；失败事件必须落日志，并在前端
  非阻塞位置展示"行情数据截止至 <上次成功时间>"提示。
- **FR-024**: 项目**首次上线**时，系统 **必须** 执行一次性**历史回填任务**，
  通过 AKShare 历史数据接口（例如 `stock_zh_a_hist()`）拉取最近 **60 个 A 股
  交易日**的全量行情数据写入 StockDaily，保证 Day-1 的"启动在即"、"主升浪"、
  自选股 5 日累计涨跌幅等所有需要历史窗口的板块立即可用。回填任务
  **必须** 在常规 11:30 / 15:15 调度启动**之前**完成且**只执行一次**
  （以"StockDaily 表非空"或独立的 `backfill_completed` 迁移标记作为幂等
  依据，避免重复执行）；回填事件作为 `type: initial-backfill` 条目记录到
  AuditLog。回填失败必须立即告警并阻止常规调度上线，以免空库状态下生成
  误导性的空筛选结果。

**金融资讯抓取与保留**

- **FR-030**: 系统 **必须** 使用财联社（CLS）电报接口（基于 AkShare
  `stock_telegraph_cls`，详见 Assumptions）抓取金融资讯；抓取频率为：
  (a) 与 11:30 / 15:15 股票数据同步抓取各一次；
  (b) 每日 21:00 额外抓取一次；
  (c) 次日 06:00 额外抓取一次。
- **FR-031**: 系统 **必须** 仅保留最近 7 天的资讯原文，超过 7 天的资讯在每次
  抓取成功后被清理；**但**被 AI 引用过的总结 / 点评 / 预测文本 **不得** 被清理。

**Dashboard 筛选：启动在即 + 主升浪**

- **FR-040**: "启动在即"列表 **必须** 使用以下条件联合筛选：
  (a) 仅个股（排除 ETF / 基金 / 指数）；
  (b) 属于 A 股主板或科创板；
  (c) 非 ST / \*ST / 退；
  (d) 总市值 > 100 亿人民币；
  (e) 当日涨幅 > 4%；
  (f) 当日成交量 > 过去 10 个交易日成交量均值；
  (g) 当日收盘价 > 近 10 日均线；
  (h) 过去 11 个交易日（可忽略最近 1 至 2 个交易日的数据）处于"区间震荡"
      状态，量化定义见 Assumptions。
- **FR-041**: "主升浪"列表 **必须** 使用以下条件联合筛选：
  (a) 仅个股；
  (b) 属于 A 股主板或科创板；
  (c) 非 ST / \*ST / 退；
  (d) 总市值 > 100 亿人民币；
  (e) 当日收盘价 > 近 5 日均线；
  (f) 当周收盘价 > 近 5 周均线；
  (g) 当日收盘价 > 近 10 日均线；
  (h) 过去 5 个交易日整体呈连续上涨状态，允许"小幅回撤"但总体趋势向上，
      量化定义见 Assumptions。
- **FR-042**: 两份列表 **必须** 各自以"当日涨跌幅降序"作为默认排序；各列表
  展示条数 **默认上限 20 条**。
- **FR-043**: 每条条目 **必须** 展示：股票名称（左）+ 当日涨跌幅百分比（右），
  颜色遵循"涨红跌绿"的 A 股惯例。

**Dashboard AI 点评三栏**

- **FR-050**: 系统 **必须** 在每日 11:30 股票数据抓取成功后 10 分钟内生成
  "今日午评"，在 15:15 抓取成功后 10 分钟内生成"今日晚评"，在 21:00 资讯额外
  抓取成功后 10 分钟内生成"未来预测"。
- **FR-051**: 三栏内容 **必须** 使用 DeepSeek LLM 作为生成引擎；生成过程中必须把
  当日市场快照（或当日资讯集合）作为上下文。
- **FR-052**: AI 生成的点评 / 预测 **必须** 永久保留，并随条目一起持久化以下
  两类时间戳：
  (a) `generated_at` —— 本次 AI 生成（LLM 调用成功落库）的本地时间；
  (b) `source_data_at` —— 生成所引用的上游数据的标称时间（例如午评引用"2026-
      04-22 11:30 午盘快照"、未来预测引用"2026-04-22 21:05 新闻批次"）。
  行情与资讯的滚动清理任务 **不得** 删除这些 AI 文本及其两类时间戳。
- **FR-053**: DeepSeek LLM 调用失败时，对应卡片 **必须** 回退到上一份成功内容并
  叠加"更新失败，已回退至上一版本"提示，并在下一调度窗口自动重试；
  回退期间卡片上展示的 `generated_at` 与 `source_data_at` **保持为上一次
  成功的值**，不被清空也不被更新为失败时刻的值。
- **FR-054**: 所有面向用户展示的 AI 生成卡片或弹窗（涵盖午评、晚评、未来预测、
  新闻 AI 总结、个股 AI 介绍、个股 AI 分析、预测推荐）**必须** 在视觉可见处
  同时展示 `generated_at`（AI 更新时间）与 `source_data_at`（参考数据时间）
  两项时间戳；展示格式 **必须** 统一（例如页脚一行小字，先 AI 更新时间再
  参考数据时间），便于用户一眼判断"这段文字基于什么时候的市场快照"。
- **FR-055**: 每一类 AI 产物 **必须** 定义自己的**新鲜度阈值**（默认值见
  Assumptions"关于 AI 产物的新鲜度阈值"一节）。当某条记录的 `source_data_at`
  距离当前时刻超出阈值时，卡片 / 弹窗 **必须** 展示显著的**过期标识**
  （例如琥珀色徽章、"已过期"文字、或"上次更新 XX 小时前"等），帮助用户
  识别内容可能已不反映最新情况；该标识 **不得** 仅以颜色为唯一语义通道
  （必须文字 + 视觉双编码，闭环宪法原则三无障碍含义）。

**Dashboard 自选股板块**

- **FR-060**: Dashboard 上的自选股板块 **必须** 展示当前登录用户的自选股、
  其当日涨跌幅、以及 5 个交易日累计涨跌幅；顺序与用户在自选管理页面配置的
  顺序一致。
- **FR-061**: 跨路由切换 **不得** 造成自选股板块的"空白重载"视觉闪烁（闭环
  宪法原则七，走系统级 Zustand store + 缓存）。
- **FR-062**: Dashboard 自选股板块 **必须** 使用**虚拟滚动（list virtualization）**
  渲染；当自选股数量 ≤ 30 时允许降级为普通滚动以节省实现复杂度，但数量 > 30
  时 **必须** 走虚拟滚动，保证滚动帧率与 DOM 节点数不随自选数量线性增长。
  自选股管理页面（用于搜索 / 拖拽 / 删除）同理，列表长度 > 30 时 **必须**
  使用虚拟滚动。

**Dashboard 预测推荐**

- **FR-070**: "预测推荐"板块 **必须** 与 FR-050 的"未来预测"生成任务共享同一份
  AI 判断，从识别出的利好**概念板块（题材板块）**中，每个板块挑选一只龙头个股，
  共展示 **最多 5 个板块**。板块口径 **必须** 使用 AkShare 概念板块体系（数据
  来自 `stock_board_concept_name_em` / `stock_board_concept_cons_em` 等接口），
  **不得** 使用申万行业或中信行业分类。
- **FR-071**: 当 AI 未能识别任何利好概念板块时，板块 **必须** 展示明确的"今日无
  显著利好方向"空态文案，不得出现占位符或错误文本。
- **FR-072**: "概念板块龙头"的挑选算法 **必须** 基于可复现的规则：在命中的
  概念板块成分股中，优先按"近 5 个交易日相对板块指数的超额收益"降序，其次
  按"总市值"降序；取第 1 名作为龙头，若该股票当日停牌或处于 ST / 退市状态，
  则跳过至下一名。

**新闻模块**

- **FR-080**: 新闻页面 **必须** 顶部固定展示一段 AI 总结文本（≤ 300 字）；
  总结在每次新闻抓取成功后更新，并在历史表中保留旧版本以备回溯。
- **FR-081**: 新闻列表 **必须** 默认展示过去 24 小时内的全部资讯，按发布时间
  倒序排列；每条资讯至少包含标题、来源、发布时间、正文摘要。

**个股 AI 弹出菜单**

- **FR-090**: 在 Dashboard 任一股票列表中单击某只个股条目，**必须** 在该条目
  附近弹出一个"右键菜单样式"的浮层，浮层必须在交互首帧内可见（闭环原则五）。
- **FR-091**: 浮层 **必须** 至少提供两个选项："AI 个股介绍"与"AI 股票分析"；
  点击任一选项 **必须** 打开一个弹窗，在弹窗内展示 AI 生成的对应文本。
- **FR-092**: 同一股票同一自然日内对同一种分析（个股介绍 / 股票分析）的重复
  请求 **必须** 命中缓存，不得重复调用 LLM。
- **FR-093**: 菜单已弹出时，点击浮层之外任意区域或按 `Esc` **必须** 关闭菜单，
  且不触发任何请求。

**界面布局与视觉（MagicUI）**

- **FR-100**: 首页 **必须** 提供不少于 3 组可选的动态背景模块，切换开关以代码
  常量或配置形式存在，切换不应涉及业务代码改动。
- **FR-101**: Dashboard 主布局 **必须** 采用 Bento Grid 分隔排列。布局在
  本期 **必须** 完全固定：所有卡片统一顺序、统一占位，**不得**支持用户
  隐藏 / 重排 / 调整尺寸；Bento Grid 仅负责在不同断点（`sm` / `md` / `lg`
  / `xl`）下对卡片做响应式布局适配，不承载拖拽或可见性开关。对应地，本期
  数据层 **不得** 引入任何 per-user 的 Dashboard 布局偏好字段（例如
  `dashboard_card_visibility` / `dashboard_card_order` 等）；需要个性化
  能力时通过独立 feature 引入。
- **FR-102**: 关键数值（主要指数、涨跌家数等） **必须** 使用 Number Ticker
  数字滚动动画呈现更新。
- **FR-103**: 各国主要指数 + 原油 / 黄金等期货价格 **必须** 以横向 Marquee
  滚动展示。
- **FR-104**: 领涨行业 / 热门行业 **必须** 以竖向 Marquee 滚动展示。
- **FR-105**: Dashboard 侧边 **必须** 存在一条可自动显示 / 隐藏的导航栏
  （桌面端 hover，移动端抽屉），用于在主模块之间跳转；Zustand 中的全局状态
  在跳转时 **不得** 被清空。

**Dashboard 实时刷新（Supabase Realtime 推送）**

- **FR-106**: 服务端每次成功完成以下任一事件后，**必须**通过 **Supabase
  Realtime** 向所有已打开 Dashboard 的客户端广播一个**分类事件**。本期事件
  全集（实现侧 EventKind 必须等于此清单的并集）：
  (a) 11:30 行情抓取完成 → `stock-daily-midday`；
  (b) 15:15 行情抓取完成 → `stock-daily-close`；
  (c) 11:30 / 15:15 / 21:00 / 06:00 新闻原文抓取入库完成 → `news-fetch-done`
      （独立于 AI 总结，使新闻列表能在原文落库瞬间局部刷新）；
  (d) 午评 / 晚评 / 未来预测的 AI 生成完成 → `ai-midday-done` /
      `ai-evening-done` / `ai-forecast-done`；
  (e) 21:00 / 06:00 新闻 AI 总结完成 → `news-summary-done`；
  (f) 预测推荐批次生成完成 → `sector-picks-done`；
  (g) 5 市场交易日历刷新完成 → `calendar-refresh-done`（保证 Dashboard 顶部
      "市场状态卡片"在日历窗口被补齐后即时更新，亦满足 FR-125 对日历来源
      的硬约束）。
  事件 payload **必须** 包含类别标识（取自上述 9 个 EventKind 之一）、
  本次事件的 `source_data_at` 与 `generated_at`（对应 FR-052 / FR-054），
  以及客户端需要 invalidate 的 board key 数组（详见 contracts/realtime-events.md）。
  **追加新事件类别**必须满足以下约束：(i) 不破坏 FR-108 反轮询禁令，即
  事件由真实服务端动作触发、非定时器伪造；(ii) payload 字段沿用同一信封；
  (iii) spec FR-106 同步更新事件全集，**禁止**仅在 contracts 单方扩展。
- **FR-107**: Dashboard 客户端收到对应事件后，**必须**在 2 秒内触发**该事件
  对应板块的局部刷新**，**不得**整页重载、**不得**清空 Zustand store 中
  的其它无关板块数据（闭环宪法原则七）。刷新过程中需满足原则五的"交互
  即时反馈"，即在数据替换之前至少展示骨架屏或顶部细进度条，**禁止**任何
  "静默替换 DOM 导致闪烁"的实现。
- **FR-108**: 本期**不得**在客户端引入任何定时轮询（`setInterval` / `setTimeout`
  循环 / `useSWR` 的 `refreshInterval` / Tanstack Query 的 `refetchInterval`
  等）作为数据刷新的常规路径；常规路径只有"Realtime 事件驱动"一条。该规则
  旨在避免 Realtime 与轮询叠加造成的重复请求与状态抖动。
- **FR-109**: Dashboard 右上角 **必须** 提供一个**手动刷新**按钮作为兜底
  入口：点击后强制让当前 Dashboard 用到的所有板块缓存失效并重新从服务端
  读取；按钮在请求期间显示 pending 态（spinner + `disabled`，闭环原则五）。
  手动刷新的节流约束：同一会话内两次手动刷新的最短间隔 **≥ 10 秒**，在冷却
  期内按钮显示 `disabled + "X 秒后可刷新"`，避免用户连点造成上游压力。
- **FR-111**: 当 Supabase Realtime 连接丢失（网络抖动 / 服务端重启）时，
  客户端 **必须** 自动尝试重连（指数退避，最长 30 秒一次）；连接恢复后
  **必须**立即触发一次"最新数据时间戳对齐"请求，若发现本地缓存中的
  `source_data_at` 落后于服务端，则自动触发一次与 FR-109 等价的全量板块
  刷新。连接长期（超过 5 分钟）未恢复时，Dashboard **必须**在顶部以小字
  展示"实时同步已断开，请手动刷新"的降级提示。

**响应式与一致性（闭环宪法原则三）**

- **FR-110**: 所有页面与非平凡组件 **必须** 在 ≤ `sm`（640 px）与 ≥ `lg`
  （1024 px）两个断点下完成设计与验证，信息层级与主操作保持一致。

**多市场休市状态与 A 股休市数据回退**

- **FR-120**: 系统 **必须** 维护 A 股、美股、港股、日股、韩股共 5 个市场的
  交易日历，能够对任意给定自然日（以 Asia/Shanghai 时区判定）判定该市场是否
  开市；交易日历覆盖窗口至少包含"过去 180 天 + 未来 30 天"，并在每次
  Dashboard 渲染之前验证当前日期是否落入覆盖窗口。
- **FR-121**: 当 A 股当日（Asia/Shanghai 所属自然日）被判定为休市时，
  Dashboard **必须** 在顶部以非阻塞横幅展示"今日 A 股休市，数据基于上一个
  交易日 <YYYY-MM-DD>"；横幅 **必须** 提供"关闭"按钮，关闭状态 **必须**
  以 **Asia/Shanghai 自然日边界** 持久化（localStorage key 形如
  `banner-dismissed-{YYYY-MM-DD}`；每天 00:00 重置；**不**绑定 A 股交易日
  历，长假期间每自然日各提醒一次）；横幅 **不得** 遮挡主内容的关键操作。
- **FR-122**: 当 A 股当日休市时，Dashboard 所有依赖 A 股行情数据的板块
  （启动在即、主升浪、自选股涨跌、AI 午评 / 晚评 / 未来预测、预测推荐、
  核心指数 Ticker、横向 Marquee 中的上证 / 深证等 A 股相关条目） **必须**
  使用"最近一个 A 股交易日"的数据；**严禁**展示空数据、"no data" 占位或
  回退到更早日期跳过一个 A 股交易日。
- **FR-123**: Dashboard **必须** 提供一张独立的"市场状态"卡片，展示 A 股、
  美股、港股、日股、韩股 5 个市场在"当前 Asia/Shanghai 自然日"的开市 /
  休市状态；排列顺序为 A 股 → 美股 → 港股 → 日股 → 韩股，每个市场一行，
  以文字（"开市" / "休市"） + 视觉标识（色点或图标）同时呈现；该卡片
  **无论** A 股是否休市，**始终**显示。
- **FR-124**: 当 A 股被判定为当日休市时，调度器 **必须** 跳过 11:30 与
  15:15 的 A 股行情抓取任务；跳过事件 **必须** 作为 `skipped: A-share
  holiday` 记录到任务审计日志；既有最近一个 A 股交易日的数据 **不得**
  被任何写入路径覆盖。
- **FR-125**: 市场状态卡片对"开市 / 休市"的判定 **必须** 严格来源于
  FR-120 的交易日历，**不得**使用"上游 API 是否返回数据为空"等启发式
  判断作为替代；若某一市场的日历暂时缺失，该行必须显示"未知"而非默认
  "开市"。

### Key Entities *(数据相关)*

- **User（用户）**：系统注册用户。属性包括唯一真实邮箱、唯一账号名、密码散列、
  邮箱确认状态、创建时间、最近登录时间。与"自选股条目"一对多关联。
  *实现侧拆为两张表：邮箱 / 密码散列 / 邮箱确认状态 / 最近登录时间由
  Supabase 内置的 `auth.users` 承担；账号名等业务字段落在 `profiles` 表
  并以 `auth.users.id` 作为外键，详见 data-model.md §1。*
- **Session（短期会话）**：登录后签发的短时凭证，记录用户、发放时间、到期
  时间；由浏览器 cookie 承载。
- **LongLivedToken（7 天免登录凭证）**：可选的长期凭证，到期时间 ≥ 7 × 24 小时，
  登出时强制失效；与 User 多对一关联。
- **InviteCode（邀请码配置）**：包含固定值 `violet-everGarden` 及"是否允许
  复用"标志；当前阶段单条记录，但建模时视为可扩展的多邀请码方案。
- **Stock（股票元数据）**：记录代码、名称、市场板块（主板 / 科创 / 其他）、
  是否 ST / 退市、所属行业、最近一次市值快照。
- **StockDaily（股票日行情）**：按股票 + 交易日一条记录，含开 / 高 / 低 /
  收 / 成交量 / 当日涨跌幅；保留最近 60 个交易日，过期滚动清理。
- **StockScreenResult（筛选结果快照，可选）**：按"交易日 + 筛选类型（launching-
  soon / main-uptrend）"缓存当日筛选产出，便于 Dashboard 秒级呈现；是否落表
  视性能评估决定。
- **WatchlistItem（自选股条目）**：用户 × 股票的自选关系，含用户自定义的
  `order_index`；跨设备一致。
- **NewsItem（资讯条目）**：财联社抓取的原始资讯，含标题、来源、发布时间、
  正文、外链；保留最近 7 天。
- **AIArtifact（AI 生成产物）**：按"类别（midday / evening / forecast /
  news-summary / stock-intro / stock-analysis）+ 主键（交易日 / 股票代码 / 批次）"
  保存生成内容、`generated_at`（AI 生成完成的本地时间）、`source_data_at`
  （所引用上游数据的标称时间，例如午盘快照时间 / 尾盘快照时间 / 新闻批次
  时间 / K 线窗口末端）、使用的上游数据哈希；永久保留。`generated_at`
  与 `source_data_at` 用于 FR-054 的双时间戳展示与 FR-055 的过期判定，**不得**
  在任何清理任务中被清空。
- **SectorPick（预测推荐条目）**：按"日期 + 板块名"保存，含对应龙头股代码与
  挑选依据摘要；与当日 forecast AIArtifact 一对多关联。
- **AuditLog（抓取 / 生成任务审计）**：记录每次调度任务的触发时间、任务类型、
  结果、失败原因，用于排查数据陈旧问题。
- **MarketCalendar（市场交易日历）**：按 `(market, date)` 唯一键标识一个市场
  在某一自然日（Asia/Shanghai 时区）的开市 / 休市状态；`market` 取值枚举
  `CN_A`、`US`、`HK`、`JP`、`KR`；覆盖窗口至少"过去 180 天 + 未来 30 天"；
  有独立的刷新任务保证窗口不过期。

## Success Criteria *(mandatory)*

### Measurable Outcomes

**可用性 / 体验**

- **SC-001**: 未登录用户访问任一受保护路由后，**在 1 秒内**被重定向回首页且
  自动展开登录弹窗（感知层面的"立刻"）。
- **SC-002**: 邀请码正确时，**注册接口（`POST /api/auth/register` → Supabase
  `signUp` 同步返回）从客户端发起到 UI 渲染"确认邮件已发送"反馈的端到端
  耗时 不超过 3 秒**——衡量对象为前端可控链路（网络 + 服务端 + 渲染），
  **不**包含邮件实际投递到收件箱的时长（受外部 SMTP 影响，仅作观察项记录、
  不计入本指标）。用户点击确认链接或完成登录后，进入 Dashboard 首屏的端到端
  耗时 **不超过 3 秒**。
- **SC-003**: Dashboard 冷启动（登录成功跳转过来）**在 2 秒内**完成首屏可见
  内容绘制（骨架屏或实际数据均可），首屏不允许出现空白超过 500 ms。
- **SC-004**: 用户在 Dashboard 与自选股页面、新闻页面之间来回切换 **不得**
  出现任何可见的"空白重载"；切换返回后，自选股与筛选列表必须在 **200 ms
  以内**从内存缓存复现。
- **SC-006**: 服务端在 11:30 / 15:15 / 21:00 / 06:00 任一调度事件成功落库后，
  已打开 Dashboard 的客户端在 **≤ 3 秒内** 接收到 Supabase Realtime 事件
  并完成对应板块的局部刷新；Realtime 断连后重连的 P95 耗时 **≤ 30 秒**；
  本期客户端代码中 **禁止** 存在任何数据层面的 `setInterval` / `refreshInterval`
  定时刷新（以静态代码扫描 + 运行时无 repeating timer 作为合规验证）。
- **SC-005**: 任何用户交互（点击、拖拽、提交、菜单呼出）**必须** 在首帧内
  给出可观察反馈；由自动化 Vitest 测试在交互发生的帧断言 pending 态或 UI
  变化（闭环原则五）。

**数据新鲜度**

- **SC-010**: 每个交易日的 11:30 与 15:15 调度窗口 **在触发后 5 分钟内** 完成
  当日全量行情抓取；若失败，Dashboard 在 1 分钟内就显示"数据截止至 <上次
  成功时间>"的降级提示。
- **SC-011**: 行情数据保留窗口保持在 **最近 60 个交易日**（允许 ±2 个交易日
  冗余），超过窗口的数据在下一次抓取成功后 10 分钟内被清理。
- **SC-012**: 资讯数据保留窗口保持在 **最近 7 天**（允许 ±1 天冗余）；被 AI
  引用过的总结 / 点评 / 预测文本 **100%** 在原文清理后仍然可回溯。
- **SC-013**: 午评 / 晚评 / 未来预测三者的生成成功率 **≥ 95%**（以每 30 日
  滚动统计），失败时自动回退到上一版本的覆盖率 **= 100%**。

**业务覆盖**

- **SC-020**: 任一交易日，"启动在即"与"主升浪"列表中 **0%** 股票来自中小板、
  创业板、北交所、ST、\*ST 或退市股票（硬性口径）。
- **SC-021**: "启动在即"与"主升浪"列表 **100%** 满足 FR-040 / FR-041 的全部
  条件，且当日涨跌幅口径与上游一致（**每次发布前从两份名单各随机抽样 ≥ 10
  只 × 1 个最近交易日**核对，偏差 ≤ 0.01%）。
- **SC-022**: 自选股板块的"当日涨跌幅"与"5 日累计涨跌幅" **100%** 可对得上
  同日 K 线收盘价计算结果（**每次发布前抽样 ≥ 30 只自选股 × 1 个最近交易日**
  核对，偏差 ≤ 0.01%）。

**安全 / 合规**

- **SC-030**: 用户密码存储形式 **100%** 为不可逆散列；任何一次代码或配置审计
  都不应在数据库或日志中出现明文密码。
- **SC-031**: "7 天免登录"凭证在登出后 **100%** 失效；从登出成功到再次用同一
  凭证访问时被拒绝的时间 **≤ 5 秒**。
- **SC-032**: 未登录用户访问受保护路由的拦截成功率 **= 100%**（不存在漏网路由）。

**响应式**

- **SC-040**: 所有首屏页面在 `sm`（≤ 640 px）与 `lg`（≥ 1024 px）两个断点下
  的"主要按钮 / 信息层级 / 主操作路径"抽查一致率 **≥ 95%**；不得出现横向
  溢出或必须依赖 hover 才能完成的关键动作。

**多市场休市状态**

- **SC-050**: A 股休市日（周末 / 法定节假日）访问 Dashboard 时，顶部横幅的
  出现率 **= 100%**；所有行情相关板块引用的交易日 **= 最近一个 A 股交易日**
  （偏差 0 天）。
- **SC-051**: 市场状态卡片对 A 股 / 美股 / 港股 / 日股 / 韩股 5 个市场"当日
  是否开市"的判定，与权威交易所公告一致率 **= 100%**（每月抽样不少于 10
  个非常规日期——跨年、春节、国庆、美国独立日、感恩节、圣诞节、日本黄金周、
  韩国秋夕、元旦等——全部命中）。
- **SC-052**: 市场交易日历的覆盖窗口始终满足"过去 ≥ 180 天 + 未来 ≥ 30 天"；
  当窗口即将不足 7 天时，自动刷新任务 **必须** 在 24 小时内完成补齐，失败
  则在 Dashboard 上显示过期提示。

**AI 产物时间戳与新鲜度**

- **SC-060**: 所有对用户展示的 AI 卡片 / 弹窗（午评、晚评、未来预测、新闻
  总结、个股介绍、个股分析、预测推荐）**100%** 同时展示 `generated_at` 与
  `source_data_at` 两类时间戳；审查时任一条缺失视为功能回归。
- **SC-061**: 当某条 AI 产物的 `source_data_at` 距离当前时刻超出对应类别的
  新鲜度阈值（见 Assumptions）时，过期标识在下一次 Dashboard 刷新内出现率
  **= 100%**；抽样核对延迟 ≤ 1 次 Dashboard 刷新周期。
- **SC-062**: LLM 调用失败触发回退时，卡片上展示的 `generated_at` 与
  `source_data_at` **100%** 等于上一次成功落库的值，不被失败尝试的时间
  覆盖（避免"假新鲜"）。

## Assumptions

**关于用户对算法的提问（直接答复 + 默认口径）**

- **财联社（CLS）接口**：用户提到的 `stock_telegraph_cls` **是** AkShare
  提供的有效接口（`ak.stock_telegraph_cls()`），返回财联社电报流式资讯；
  默认在不需要 `symbol` 参数下调用即可拉取最新一批，再按发布时间过滤。
  本 spec 以此为默认；若 Plan 阶段研究发现更合适的 AkShare 接口（例如
  `news_cctv`、`stock_info_global_cls`），可在 plan.md / research.md
  中提出替换并回写到此处。
- **"启动在即"的"区间震荡"量化定义**：原描述"前 11 交易日处于区间震荡"偏
  定性。本 spec 给出可测量的默认口径：取 `T-12` 至 `T-2`（共 11 个交易日，
  允许忽略最近 1 至 2 个交易日，即窗口可选择以 `T-1` 或 `T-2` 为右端，
  向前回看 11 个交易日）的收盘价序列，满足以下全部条件视为"区间震荡"：
  (i) 窗口振幅 `(max − min) / mean ≤ 15%`；
  (ii) 窗口内最大单日涨跌幅（绝对值） `≤ 5%`；
  (iii) 对 log(close) 做一元线性回归，斜率绝对值 `≤ 0.3% / 交易日`
       （即总体不具备明显趋势）。
  这组阈值与 FR-040 的其它硬条件联合使用；阈值在 Plan 阶段可调但不超出
  "±30% 相对基线"。
- **"主升浪"的"连续上涨（可忽略小幅回撤）"量化定义**：取 `T-5` 至 `T-1` 共
  5 个交易日的收盘价序列，满足以下全部条件视为"连续上涨":
  (i) 窗口整体收益 `(close[T-1] / close[T-5]) − 1 ≥ 3%`；
  (ii) 窗口内日收益为正的交易日数 `≥ 3`；
  (iii) 窗口内单日最大回撤（即 `(close[t] − close[t-1]) / close[t-1]` 的
       最小值） `≥ −3%`（允许存在 ≤ 3% 的单日回撤，更深者视为破坏趋势）。
- **"主升浪筛选条件中重复的均线约束"**：FR-041 同时保留"日线 > 5 日均线"
  与"当日收盘价 > 10 日均线"。这并非冗余——前者确保短期动量，后者排除
  在 10 日内刚刚见顶回落的股票；保留两者更严格。
- **市值口径**：FR-040 / FR-041 中"市值 > 100 亿"默认指 **总市值** 而非
  流通市值；Plan 阶段若发现上游接口仅提供流通市值，可改为"流通市值 ≥
  100 亿"并在此处回写。
- **成交量对比基线**：FR-040 中"当日成交量 > 过去 10 个交易日均值"中，10
  日均值计算**不包括**当日成交量。

**关于身份与访问**

- 邀请码当前仅有一枚硬编码值 `violet-everGarden`，按"不允许泄漏到客户端
  bundle"的默认要求处理（校验在服务端进行）。
- 账号名视为唯一，并对大小写不敏感（`Alice` 与 `alice` 不可同时存在）；
  入库前以小写形式规范化。字符集限定为 ASCII 字母 / 数字 / `_` / `-`，
  长度 3~20（正则 `^[A-Za-z0-9_-]{3,20}$`；见 FR-008）。账号名仅作为站内显示名，
  不作为 Supabase Auth 登录凭证。
- 密码长度默认要求 ≥ 8 位，包含字母 + 数字即可，暂不强制特殊字符；该默认
  值可被 `/speckit.clarify` 调整。
- 所有受保护路由的判定以"未读取到有效 session 或 7 天长期凭证"为未登录；
  本期启用邮箱确认；第三方 OAuth、短信验证与忘记密码暂不实现。
- **认证库的选型**：本项目 **必须** 使用 Supabase Auth（由宪法 v1.2.0
  「技术栈与约束」段作为硬约束指定）。本期邮箱 + 密码流程直接使用用户真实邮箱
  作为 Supabase Auth 的 email 主标识，并开启 Confirm email；真实账号名保存在
  业务层 `profiles` 表作为唯一显示名与资料键。Supabase Auth 承担"邮箱确认 +
  签发 session JWT + 与 RLS 联动"的角色。7 天免登录通过 Supabase session 的
  `expiresIn` / refresh token 策略实现。
- **被拒绝的备选方案：NextAuth（Auth.js）**。NextAuth 技术上能够覆盖本 spec
  的所有身份功能需求（`CredentialsProvider` 支持纯邮箱密码、自定义
  `signIn` 回调可做邀请码校验、JWT 会话可实现 7 天免登录、middleware 可
  拦截未登录路由），但与当前项目**不契合**，因此本期不予采用：
  (i) **冲突宪法硬约束**—— 宪法 v1.2.0 明确 `默认使用 Supabase Auth`，
      偏离需走宪法修订流程；直接在 spec 引入 NextAuth 会让 `/speckit.plan`
      的 Constitution Check 失败；
  (ii) **RLS 集成代价高**—— Supabase RLS 原生认 Supabase 自签的 JWT，
       换成 NextAuth 自签 JWT 后要么重做一层"NextAuth → Supabase JWT"
       的签名转换（复杂且脆弱），要么绕过 RLS 走 `service_role` key（直接
       违反宪法「所有对客户端暴露的表必须定义 RLS」的要求）；
  (iii) **重复用户模型**—— Supabase Auth 自带 `auth.users`；NextAuth 默认
        还要引入 `users` / `accounts` / `sessions` 三张表，两套实体强制
        对齐会让 Drizzle schema 冗余；
  (iv) **边际收益小**—— 本期明确砍掉了 OAuth、magic link、密码找回等
       功能（见 Q3 / Q4 clarify 结论），NextAuth 真正替代的仅是"签 session
       JWT"这一单点能力，而 Supabase Auth 在签 JWT 的同时还天然与 RLS 联动。
  若未来确需 NextAuth（例如引入多方 OAuth 提供商），需先走宪法修订流程，
  把「认证与授权」条目从"默认使用 Supabase Auth"放宽为允许并存，版本至少
  升至 MINOR，同时给出 NextAuth 与 RLS 的集成方案。

**关于调度与时区**

- 所有定时任务以 **Asia/Shanghai** 时区为准；交易日判定参考上交所 / 深交所
  公开的交易日历；节假日不触发调度。
- 11:30 与 15:15 为触发时刻的下限，实际允许 `±5 分钟` 抖动；`21:00` 与
  `06:00` 允许 `±10 分钟` 抖动。

**关于多市场交易日历（US12 / FR-120～FR-125 的数据来源）**

- **A 股日历**：默认使用 AkShare 的 `tool_trade_date_hist_sina()` 或
  `calendar_tradingdate_sse`（由 Plan 阶段 research 选定更稳定的那一个），
  以此判定 A 股当日是否开市。
- **美股 / 港股 / 日股 / 韩股日历**：本 spec 默认优先使用同样通过 Sealos 代理
  的第三方日历库（例如 `pandas-market-calendars`，内置 NYSE / HKEX / JPX /
  KRX 的交易日历）作为单一数据源；若 Plan 阶段发现 AkShare 直接提供对应市场
  的日历接口，则优先使用 AkShare 以保持数据源一致性。
- **"当日"判定基准**：统一以 **Asia/Shanghai 所属自然日** 判断任一市场当日
  是否开市；这意味着用户所处的自然日与本地判断一致，不会因用户 VPN / 设备
  时区而浮动。
- **日历覆盖窗口**：至少"过去 180 天 + 未来 30 天"；实际实现可扩到未来
  365 天。窗口不足 7 天时由独立的日历刷新任务补齐。
- **本期 5 个市场为硬约束**：市场状态卡片仅覆盖 A 股、美股、港股、日股、
  韩股。欧洲 / 印度 / 澳洲 / 新加坡 / 中国台湾等市场视为 P3 之后的增量功能，
  不在本次范围。

**关于数据量与性能**

- 全量 A 股股票数量按约 5,500 只规模估算；60 个交易日行情条目约
  5,500 × 60 ≈ 33 万条，单机数据库完全可承载。
- 每日财联社电报条目按 500 条估算，7 日约 3,500 条。
- "启动在即" / "主升浪"筛选结果在每日 15:15 抓取后 5 分钟内预计算并缓存，
  Dashboard 请求时直接读缓存，避免每位用户请求时都走全量扫描。

**关于 AI 引擎**

- 本版本 AI 能力**仅使用 DeepSeek LLM**（DeepSeek API，OpenAI 兼容端点）。
  任何接口失败按 FR-053 规则回退到上一版本（`generated_at` / `source_data_at`
  保持为上一次成功的值）；暂不考虑多厂商路由或降级到其它模型。
- LLM 输入 **不得** 包含用户密码、长期凭证等敏感字段；调用前在服务端构造
  prompt，客户端不直接调用 DeepSeek。

**关于 Dashboard 实时推送与轮询禁令（FR-106~FR-111）**

- 本期采用 **Supabase Realtime** 作为 Dashboard 数据更新的唯一"主动通道"，
  服务端在抓取 / AI 生成完成后以事件方式通知前端，前端收到事件后**仅刷新
  对应板块**（不整页重载）。选择 Realtime 的原因：本产品的写入事件频率极
  低（每日 5~6 次离散触发），短连接轮询会在大部分时间空转；而宪法已允许
  Supabase Realtime 作为"按需启用"能力，不会引入新依赖。
- 客户端 **禁止** 使用任何形式的数据层定时刷新（见 FR-108）。这是一条明确
  的反模式禁令：即便在 plan 阶段看起来"加个 refreshInterval 省事"，也必须
  走 Realtime；否则会造成 Realtime 事件与轮询请求叠加，触发不必要的上游
  请求和 Zustand 状态抖动。**例外**：**手动刷新按钮**（FR-109）属于"用户
  主动触发"，不计入此禁令。
- Realtime 频道的 RLS 规则需要保证 `data-updated` 事件对所有已登录用户都
  可订阅（事件 payload 中不含敏感字段，仅含事件类别与时间戳），符合宪法
  "所有对客户端暴露的表必须定义 RLS"的约束。具体 channel / topic 命名与
  订阅代码结构由 Plan 阶段产出。

**关于板块粒度（US8 / FR-070～FR-072）**

- "未来预测"与"预测推荐"中的"板块"统一采用 **概念板块 / 题材板块** 口径
  （经 Clarify 确认，2026-04-22）。理由：题材板块与资讯事件的对应关系更
  紧密，更能反映"消息面 → 利好方向"的因果链；申万一级（31 个）粒度偏粗，
  会把"AI 算力"这种精准题材泛化成"计算机"，丢失信息量。
- 板块元数据与成分股来自 AkShare 概念板块接口，例如
  `stock_board_concept_name_em`（板块清单）、`stock_board_concept_cons_em`
  （指定板块的成分股列表）、`stock_board_concept_index_ths`（板块指数行情）。
  Plan 阶段可根据接口稳定性在同类接口间替换，但口径必须始终是"概念板块"。
- AI Prompt 工程需要把概念板块清单（含 Board 代号与名称，如
  `BK0855 虚拟现实`）作为上下文注入 LLM，并约束 LLM 的输出必须从清单中
  挑选 —— **禁止**让模型自创板块名称，避免"幻觉板块"。

**关于 AI 产物的新鲜度阈值（FR-055 默认值）**

每一类 AI 产物对应的新鲜度阈值如下，以 `source_data_at` 与当前时刻之差为
判定依据。超过阈值即触发 FR-055 的过期标识：

| 类别 (kind)        | 新鲜度阈值（默认） | 判定依据 |
|-------------------|------------------|---------|
| midday（午评）     | 当日 15:15 之前；超过 24 小时硬性过期 | 尾盘后理应有晚评接替；超出 24h 说明调度异常 |
| evening（晚评）    | 次交易日 11:30 之前；超过 24 小时硬性过期 | 下一交易日的午评会接替它 |
| forecast（未来预测）| 24 小时 | 每日 21:00 与次日 06:00 各生成一次 |
| news-summary（新闻总结）| 6 小时 | 每天至少 4 批抓取（11:30 / 15:15 / 21:00 / 06:00） |
| stock-intro（个股介绍）| 30 天 | 行业与主营变化频率低 |
| stock-analysis（个股分析）| 1 个交易日 | 依赖最新 K 线，次日即过期 |

以上阈值为 Plan 前的默认值，允许在 `/speckit.clarify` 或 Plan 阶段微调（不
超出 "±50% 相对基线"）。判定 A 股休市日时，"次交易日"按 A 股交易日历回推
而非按自然日。

**关于 MagicUI / 第三方组件**

- MagicUI 的 MagicCard、Number Ticker、Marquee（横 / 竖）、Bento Grid
  均已通过 MCP 连接到本仓库（用户在描述中声明），Plan 阶段可直接使用。
- 除 MagicUI 列出的以上组件和 shadcn/ui 默认集合外，暂不引入其它 UI 库
  （闭环宪法原则四）。

**关于当前作用域之外的事项**

- 短信验证、第三方登录（Google / 微信 / GitHub）与忘记密码暂不在本次交付
  范围；未来若纳入需走宪法 / 宪法之下的新 feature 流程。邮箱确认属于本期范围。
- 付费订阅、交易执行、实盘下单功能 **显式不在本次范围**；本产品是数据
  分析与辅助决策工具，不是交易通道。
- 管理后台（新增 / 撤销邀请码、封禁用户、手动重跑任务）作为独立 feature
  在本次之后交付。
- **用户自助注销账号**与**数据导出**（自选股 CSV / 完整 GDPR 风格导出）
  **显式不在本期范围**（经 Clarify 确认，2026-04-22）。理由：本产品是邀请码
  受控的小圈子部署，无 GDPR / CCPA 辖区压力；用户注销与数据清理延后到后续
  "管理后台"feature，由管理员入口承担。本期如有个别用户注销请求，按手工
  DB 操作处理。
